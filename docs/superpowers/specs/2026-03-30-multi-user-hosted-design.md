# Multi-User Hosted Mode

## Goal

Allow multiple users to use the app simultaneously on a shared hosted server
(no-camera mode) without interfering with each other. No login required, no
files stored on the backend filesystem.

## Context

Currently the app has a single shared `FrameStore` that holds one frozen frame.
All detection, inspection, alignment, and sub-pixel endpoints read from this
shared store. If two users load different images, they overwrite each other's
frames and get wrong results.

The good news: **all annotation and measurement state is already client-side.**
Each browser tab has its own `state.js`, localStorage, calibration, DXF
overlay, and inspection results. The only shared server-side state that causes
conflicts is the frozen frame.

### What changes in hosted mode

| Component | Current (single-user) | Hosted (multi-user) |
|-----------|----------------------|---------------------|
| Camera | Real camera or NullCamera | Always NullCamera (no-camera mode) |
| Frame store | Single global `FrameStore` | Per-session `FrameStore` |
| Config | Shared `config.json` on disk | Read-only defaults; per-user settings stay client-side |
| Snapshots | Saved to filesystem | Disabled (no server filesystem writes) |
| Session save/load | Download/upload JSON files | Same (client-side download/upload, no server storage) |
| MJPEG stream | Live camera feed | Not available (no camera) |
| Annotations | Client-side `state.js` | Same (already isolated) |

## Design

### 1. Session management

**Session ID**: Random UUID generated by the frontend on first page load. Stored
in `sessionStorage` (not `localStorage` — unique per tab, survives page refresh
within the tab, cleared when tab closes). Sent as an `X-Session-ID` header on
every API request.

```js
// Frontend: generate or retrieve session ID
if (!sessionStorage.getItem("sessionId")) {
  sessionStorage.setItem("sessionId", crypto.randomUUID());
}
const SESSION_ID = sessionStorage.getItem("sessionId");

// Add to every fetch call via a wrapper or interceptor
function apiFetch(url, options = {}) {
  options.headers = { ...options.headers, "X-Session-ID": SESSION_ID };
  return fetch(url, options);
}
```

No login, no cookies, no server-side session creation endpoint. The session ID
is just a key into the per-session frame store map.

### 2. Per-session frame store

Replace the single global `FrameStore` with a `SessionFrameStore` that manages
multiple sessions.

```python
import re
_UUID_RE = re.compile(r"^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$")
DEFAULT_SESSION_ID = "_default"  # used in single-user camera mode

class SessionFrameStore:
    """Thread-safe per-session frame storage with auto-expiration."""

    def __init__(self, max_sessions: int = 50, ttl_seconds: int = 1800):
        self._frames: dict[str, tuple[np.ndarray, float]] = {}
        self._lock = threading.Lock()
        self._max_sessions = max_sessions
        self._ttl = ttl_seconds

    def store(self, session_id: str, frame: np.ndarray) -> None:
        with self._lock:
            self._evict_expired()
            if len(self._frames) >= self._max_sessions and session_id not in self._frames:
                raise RuntimeError("Too many active sessions")
            self._frames[session_id] = (frame.copy(), time.monotonic())

    def get(self, session_id: str) -> np.ndarray | None:
        with self._lock:
            self._evict_expired()  # also evict on get, not just store
            entry = self._frames.get(session_id)
            if entry is None:
                return None
            frame, _ = entry
            self._frames[session_id] = (frame, time.monotonic())  # touch
            return frame.copy()

    def clear(self, session_id: str) -> None:
        """Remove a specific session's frame."""
        with self._lock:
            self._frames.pop(session_id, None)

    def has_frame(self, session_id: str) -> bool:
        with self._lock:
            return session_id in self._frames

    def _evict_expired(self) -> None:
        now = time.monotonic()
        expired = [k for k, (_, ts) in self._frames.items() if now - ts > self._ttl]
        for k in expired:
            del self._frames[k]
```

**Frame copies**: Both `store()` and `get()` copy the numpy array. This
matches the current `FrameStore` behavior and prevents concurrent requests
from mutating each other's frame data.

**Eviction on both store and get**: Prevents stale sessions from consuming
memory when no new sessions are being created.

**Memory budget**: Each frame is ~15MB (2592x1944x3 bytes). 50 sessions = ~750MB
max. Configurable via environment variable `MAX_SESSIONS`.

**Expiration**: Sessions expire after 30 minutes of inactivity (no API calls
with that session ID). The `_evict_expired` check runs on every `store()` and `get()` call.

### 3. Backend API changes

**Session ID extraction**: A FastAPI dependency that reads `X-Session-ID` from
the request headers, validates UUID format, and falls back to `DEFAULT_SESSION_ID`
in single-user camera mode:

```python
def get_session_id(request: Request) -> str:
    session_id = request.headers.get("X-Session-ID")
    if not session_id:
        if hosted_mode:
            raise HTTPException(400, detail="Missing X-Session-ID header")
        return DEFAULT_SESSION_ID  # single-user camera mode
    if not _UUID_RE.match(session_id):
        raise HTTPException(400, detail="Invalid session ID format")
    return session_id
```

**Upload size limit**: In hosted mode, add a max upload size check to
`/load-image`. FastAPI's `UploadFile` streams by default, so read with a cap:

```python
MAX_UPLOAD_BYTES = 20 * 1024 * 1024  # 20MB

async def load_image(file: UploadFile, session_id: str = Depends(get_session_id)):
    data = await file.read(MAX_UPLOAD_BYTES + 1)
    if len(data) > MAX_UPLOAD_BYTES:
        raise HTTPException(413, detail="Image too large (max 20MB)")
    ...
```

**Session cleanup on tab close**: Add a `DELETE /session` endpoint for
best-effort cleanup via `beforeunload`:

```python
@router.delete("/session")
async def delete_session(session_id: str = Depends(get_session_id)):
    frame_store.clear(session_id)
    return {"ok": True}
```

Frontend calls this in `beforeunload` (best-effort — may not complete):
```js
window.addEventListener("beforeunload", () => {
  navigator.sendBeacon("/session", new Blob(
    [JSON.stringify({})],
    { type: "application/json" }
  ));
});
```
Note: `sendBeacon` doesn't support custom headers, so the session deletion
endpoint should also accept the session ID from a cookie or query param as
a fallback. Or use a `fetch` with `keepalive: true`.

**Endpoints that use frame_store** — all need the session ID:

| Endpoint | Current | Change |
|----------|---------|--------|
| `POST /freeze` | `frame_store.store(frame)` | Skip (no camera in hosted mode) |
| `GET /frame` | `frame_store.get()` | `frame_store.get(session_id)` |
| `POST /load-image` | `frame_store.store(frame)` | `frame_store.store(session_id, frame)` |
| `POST /detect-edges` | `frame_store.get()` | `frame_store.get(session_id)` |
| `POST /detect-circles` | `frame_store.get()` | `frame_store.get(session_id)` |
| `POST /detect-lines` | `frame_store.get()` | `frame_store.get(session_id)` |
| `POST /detect-lines-merged` | `frame_store.get()` | `frame_store.get(session_id)` |
| `POST /detect-arcs-partial` | `frame_store.get()` | `frame_store.get(session_id)` |
| `POST /preprocessed-view` | `frame_store.get()` | `frame_store.get(session_id)` |
| `POST /inspect-guided` | `frame_store.get()` | `frame_store.get(session_id)` |
| `POST /align-dxf-edges` | `frame_store.get()` | `frame_store.get(session_id)` |
| `POST /fit-feature` | Uses points, not frame | No change |
| `POST /refine-point` | `frame_store.get()` | `frame_store.get(session_id)` |
| `POST /gradient-overlay` | `frame_store.get()` | `frame_store.get(session_id)` |
| `POST /match-dxf-lines` | No frame_store usage | No change |
| `POST /match-dxf-arcs` | No frame_store usage | No change |
| `POST /load-snapshot` | `frame_store.store(frame)` | Disabled in hosted mode (returns 403) |
| `DELETE /session` | N/A | New: clears session frame |

**Backward compatibility**: When `no_camera` is false (local single-user mode
with a real camera), the session ID header is optional. If missing, use a
default session ID `"_default"`. This means the single-user camera workflow is
unchanged — no session management needed when running locally.

### 4. Config in hosted mode

`config.json` becomes read-only in hosted mode. The app loads defaults at
startup and serves them, but `POST /config/ui` and `POST /config/tolerances`
become no-ops (return current values without writing).

Settings that matter per-user (tolerances, sub-pixel method, snap radius) are
already stored in the frontend `state.settings` object. The backend config
endpoints just provide initial defaults on page load.

A new config flag: `"hosted": true` in `config.json` or `HOSTED=1` env var.
When set:
- `no_camera` is forced to `true`
- Config write endpoints return 403 with `{"detail": "Read-only in hosted mode"}`
- Snapshot save/load is disabled
- `SessionFrameStore` is used instead of `FrameStore`

### 5. Disabled features in hosted mode

These features rely on server filesystem or camera hardware:

| Feature | Why disabled | Frontend behavior |
|---------|-------------|-------------------|
| `/freeze` | No camera | Button hidden (already hidden in no-camera mode) |
| `/snapshot` | Writes to filesystem | Button hidden |
| `/snapshots`, `/load-snapshot` | Reads filesystem | Endpoints return empty/404 |
| `POST /config/ui`, `/config/tolerances` | No filesystem writes | Return current defaults, ignore writes |
| Camera dropdown | No camera | Hidden entirely |
| `/stream` | No camera | Not requested (img element hidden) |

### 6. Frontend changes

**Minimal changes needed:**

1. **Session ID on all API calls**: Replace all `fetch()` calls with a wrapper
   that adds the `X-Session-ID` header. Or use a global fetch interceptor.

2. **Hide camera-specific UI when hosted**: The app already hides camera
   controls when `no_camera` is true. The Camera dropdown should also hide.

3. **Image load is the primary workflow**: In hosted mode, users drag-drop or
   load images. The "Load image" button becomes the main entry point.

4. **No auto-save to server**: Auto-save already uses `localStorage` (per-tab).
   Session save/load uses client-side JSON download/upload. No change needed.

### 7. Deployment

**Docker**: The existing Docker container works. Set `HOSTED=1` env var and
`NO_CAMERA=1`. Expose port 8000.

```bash
docker run -e HOSTED=1 -e NO_CAMERA=1 -p 8000:8000 microscope
```

**Scaling**: The app is stateless except for the in-memory frame store. For
horizontal scaling, you'd need sticky sessions (route same session ID to same
instance). But for 50 concurrent users, a single instance with ~1GB RAM is
sufficient.

**CORS**: If hosted on a different domain, add CORS middleware. Currently the
frontend and backend are same-origin (served from the same Uvicorn instance).

### 8. What this does NOT change

- Frontend state management (already client-side, already isolated)
- Annotation data model
- DXF parsing, detection algorithms, guided inspection, sub-pixel refinement
- Session save/load format (JSON files, client-side)
- PDF/CSV/DXF export (generated client-side or from the per-session frame)
- Undo/redo (client-side)

### 9. Security considerations

- **No authentication**: Anyone with the URL can use the app. Acceptable for
  a demo/tool-sharing scenario. Add authentication later if needed.
- **Session ID guessing**: UUIDs are 128-bit random — infeasible to guess.
  One user cannot access another's frozen frame.
- **Memory DoS**: Max sessions cap (50) prevents one client from creating
  unlimited sessions. Frame size is bounded by image upload (add a max file
  size limit, e.g., 20MB).
- **No server-side data persistence**: Nothing is written to disk. When the
  server restarts, all frame stores are cleared. Users' work is preserved in
  their browser's localStorage and downloaded session files.
